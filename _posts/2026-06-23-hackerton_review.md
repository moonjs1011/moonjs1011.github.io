100개 이상의 지원 팀 가운데 본선 6개 팀에 선정되어 카카오 AI 캠퍼스에서 약 30시간 동안 해커톤을 진행했다. 우리 팀은 **AI 기반 보이스피싱 통신서비스 공동 대응 플랫폼**을 주제로 선택했고, 나는 Spring Boot 백엔드 개발을 담당했다.

![해커톤 팀 작업 현장](/assets/img/hackerton/KakaoTalk_Photo_2026-06-23-21-45-32%20018.jpeg)

## 맡은 역할

백엔드 서버는 Flutter 앱과 운영 대시보드, AI 모델 사이의 요청을 중계하는 역할을 맡았다. 제한된 시간 안에 다음 작업을 진행했다.

- Spring Boot 기반 REST API 설계 및 구현
- PostgreSQL 스키마와 핵심 엔티티 설계
- Flutter 앱·대시보드와의 API 연동
- AI 모델 요청 및 응답을 전달하는 서버 흐름 구성

초기에는 빠른 MVP 구현을 위해 NoSQL도 검토했다. 그러나 향후 AI 모델 고도화 과정에서 임베딩 검색과 관계형 데이터 관리가 함께 필요할 가능성을 고려해 PostgreSQL을 선택했다. 실제 해커톤에서는 Vector DB 기능까지 사용하지 않았지만, 확장 가능성과 팀의 활용 경험을 기준으로 선택 근거를 명확히 했다는 점에 의미가 있었다.

![해커톤 개발 공간](/assets/img/hackerton/15.jpeg)

## 가장 어려웠던 지점

기술 자체보다 여러 클라이언트와 AI 파트가 동일한 요청 흐름을 이해하도록 맞추는 일이 더 어려웠다. API 명세와 데이터 흐름 문서가 Mattermost, 카카오톡, Discord, Notion에 분산되면서 최신 문서를 찾는 시간이 늘었고, 이미 합의한 내용을 반복해서 확인하는 상황이 발생했다.

마감이 가까워질수록 ERD 수정, API 구현, 연동 설명이 동시에 몰렸다. 이 경험을 통해 백엔드 개발자의 역할에는 코드 작성뿐 아니라 **팀이 같은 시스템 모델을 공유하도록 만드는 일**도 포함된다는 점을 배웠다.

![팀 개발 진행 모습](/assets/img/hackerton/7.jpeg)

## 결과와 배운 점

수상에는 이르지 못했지만 실제로 동작하는 MVP를 완성했다. 한편 해커톤에서는 구현 완성도만큼 문제 정의, 사용자 가치, 발표의 전달력도 평가 결과에 큰 영향을 준다는 점을 확인했다.

다음 프로젝트에서는 아래 원칙을 적용하려 한다.

1. 문서의 단일 기준 저장소를 정하고 최신 상태를 명확히 표시한다.
2. 개발 시작 전에 요청·응답 흐름과 역할 경계를 한 장의 다이어그램으로 합의한다.
3. 기능 구현과 발표 준비의 마감 시간을 분리한다.
4. 기술 선택은 가능한 기능이 아니라 현재 요구사항과 검증 비용을 기준으로 결정한다.

짧은 시간 동안 팀원들과 기술적 선택을 논의하고 동작하는 결과를 만든 경험은 백엔드 개발을 계속하고 싶은 이유를 다시 확인하게 했다. 함께 완주한 팀원들에게 감사한다.
